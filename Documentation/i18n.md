# i18n Audit — Remaining Offenders

Last updated: 2026-03-02

This document tracks all user-facing interfaces in LAMB that are missing internationalization (i18n) support. Items are grouped by priority level based on user visibility and impact.

- **RubricMetadataForm.svelte** — Added i18n to all ~20 hardcoded strings

## P0 — Critical (entire pages/flows with zero i18n)

### 1. Organization Admin Page (`org-admin/+page.svelte`)

- **File**: `frontend/svelte-app/src/routes/org-admin/+page.svelte`
- **Size**: ~4000+ lines
- **Issue**: Entire page has zero i18n — all strings are hardcoded English
- **Estimated strings**: 200+ (section headers, form labels, button text, status messages, error messages, table headers, tooltips, modal text)
- **Sections affected**:
  - Organization settings / configuration
  - Member management (invite, roles, remove)
  - Organization features/limits management
  - LTI configuration
  - All modals and forms within the page
- **Effort**: High — largest single-page remediation in the app

### 2. LTI Jinja2 Templates (backend-served HTML)

- **Directory**: `backend/creator_interface/templates/`
- **Files** (6 templates):
  - `lti_redirect.html` — LTI launch redirect page
  - `lti_creator_redirect.html` — Creator-specific redirect
  - `lti_error.html` — Error display page
  - `lti_debug.html` — Debug information display
  - `lti_teacher_launcher.html` — Teacher's assistant launcher/selector
  - `lti_student_chat.html` — Student chat interface
- **Issue**: Zero i18n infrastructure. All templates contain hardcoded English strings. No Jinja2 i18n extension or `gettext` integration. No locale detection from `launch_presentation_locale` LTI parameter.
- **Estimated strings**: ~82 across all templates
- **Challenge**: Requires adding a backend i18n framework (e.g., `python-i18n`, `babel`, or Jinja2's `i18n` extension), detecting locale from the LTI launch parameters, and creating translation catalogs
- **Effort**: High — needs architectural decision on backend i18n approach

## P2 — Backend API Error Messages

### 3. Inline HTML Error Responses in LTI Router

- **File**: `backend/creator_interface/lti_router.py`
- **Issue**: ~20+ inline HTML error responses with hardcoded English text returned directly from route handlers (e.g., `HTMLResponse(content="<h1>Error</h1><p>Invalid LTI request</p>")`)
- **Strings include**: Error titles, descriptive paragraphs, and instructional text for LTI launch failures
- **Note**: These are visible to end-users (students and teachers) when LTI launches fail

### 4. Backend API Error `detail` Strings

These are returned in JSON error responses (e.g., `HTTPException(detail="...")`) and surfaced by the frontend in toast notifications or error displays.

| File | Approx. strings | Examples |
|------|-----------------|----------|
| `backend/creator_interface/auth_context.py` | ~10 | "Not authenticated", "Session expired" |
| `backend/creator_interface/main.py` | ~5 | "Service unavailable" |
| `backend/creator_interface/assistant_router.py` | ~15 | "Assistant not found", "Name already exists" |
| `backend/creator_interface/knowledges_router.py` | ~15 | "Knowledge base not found", "Upload failed" |
| `backend/creator_interface/organization_router.py` | ~20 | "Organization not found", "Insufficient permissions" |
| `backend/creator_interface/kb_server_manager.py` | ~10 | "KB server connection failed" |
| `backend/lamb/completions/main.py` | ~10 | "Model not available", "Rate limit exceeded" |
| Other routers (rubrics, templates, sharing, etc.) | ~20 | Various CRUD error messages |

- **Total**: ~100+ hardcoded error strings across the backend
- **Challenge**: Backend API errors are typically returned as structured JSON. Frontend could either:
  - (a) Map error codes/keys to localized strings on the frontend, or
  - (b) Accept a locale header and return localized messages from the backend
- **Recommendation**: Option (a) is simpler and consistent with the frontend-driven i18n approach. Define error code constants in the backend, map them to i18n keys in the frontend.

## Approach Recommendations

### For P0 items:

1. **org-admin page**: Apply the same `svelte-i18n` pattern used in the rest of the app (`$locale ? $_('key', { default: 'Fallback' }) : 'Fallback'`). Add keys to all 4 locale files. This is a large but straightforward task.

2. **LTI templates**:
   - Detect locale from `launch_presentation_locale` LTI parameter (already available in the launch data)
   - Add Jinja2 i18n extension or a simple translation dict approach
   - Create translation files for the ~82 strings
   - This needs a design decision before implementation

### For P2 items:

3. **LTI router HTML errors**: Convert inline HTML to templates (or use existing `lti_error.html` template) and apply same i18n approach as LTI templates above.

4. **Backend API errors**:
   - Define error code constants (e.g., `ASSISTANT_NOT_FOUND`, `INSUFFICIENT_PERMISSIONS`)
   - Return codes in API responses alongside English default messages
   - Frontend maps codes to localized strings via existing `svelte-i18n` system
   - Incremental: can be done per-router without breaking changes
